SMCKit — Code Review

Generated: 2026-07-12  ·  Scope: Whole repository (project sources under SMCKit/, smc-dump/, SMCKitTests/). The vendored Submodules/xcconfig submodule was treated as an external dependency and not reviewed in depth.

SMCKit is a small, focused macOS framework that reads keys from the System Management Controller and decodes their raw bytes into native values, plus a smc-dump command-line tool. The code is clean, well organized (one type per file), thoroughly documented with HeaderDoc/SwiftDoc, and now covered by Swift Testing unit tests. It builds successfully (SMCKit scheme, Debug). The overall quality is good.

The most important issue is a value-decoding correctness bug: the flt and ioft floating-point SMC types are decoded from the wrong byte order, so their value comes out byte-swapped and meaningless on real hardware — while the neighbouring integer and sp78 types are handled correctly. This matters because flt/ioft are common for temperature and power sensors on Apple Silicon, and the unit tests exercise the standalone helpers but never the value(for:type:) path for these types, so the bug is invisible to the suite. Beyond that, there is a thread-safety gap on the process-wide shared instance and a handful of low-severity robustness, leak, and style items.

Notes

  • Byte-order model. SMC.m -readSMCKey:buffer:maxSize:keyInfo: (lines 528–538) reverses every value's bytes from the SMC's big-endian ordering into little-endian before handing them to Swift (with kSMCKeyACID as the sole exception). Understanding this reversal is central to judging the decoders in SMCHelper, and it is the crux of finding 1.
  • Standalone helpers vs. the value() path. The public numeric helpers (uint16, float32, ioFloat, …) are documented and tested as big-endian decoders and are internally consistent. The bug is specifically in how value(for:type:) combines them with the reversed buffer, which the tests do not cover.
  • Build. Verified with xcodebuild -project SMCKit.xcodeproj -scheme SMCKit -configuration Debug build → BUILD SUCCEEDED. The no-argument call SMC.shared.readAllKeys() in SMCDump.swift compiles because Swift supplies an implicit nil default for the trailing nullable block parameter.

Findings

High

1. flt and ioft values are decoded with the wrong byte order

Category: correctness / stability  ·  Location: SMCKit/SMCHelper.swift:78–80, interacting with SMCKit/SMC.m:528–538

Description

SMC.m reverses each value's bytes into little-endian before they reach Swift. Accordingly, the integer cases in value(for:type:) compensate with .byteSwapped: 

case "ui16": return self.uint16( data: data ).byteSwapped   // undoes the reversal → correct
case "ui32": return self.uint32( data: data ).byteSwapped
...
case "flt ": return self.float32( data: data )              // NO compensation
case "ioft": return self.ioFloat( data: data )              // NO compensation
case "sp78": return self.sp78(    data: data )              // handled internally (correct)

float32(data:) and ioFloat(data:) both read the buffer big-endian (via uint32/uint64, which assemble MSB-first). Given a buffer that SMC.m has already reversed to little-endian, they therefore operate on the byte-swapped bit pattern and are never corrected. Worked example for a temperature of 1.0 (flt): 

SMC big-endian bytes : 3F 80 00 00        (IEEE-754 1.0f)
after SMC.m reversal : 00 00 80 3F        (little-endian, stored in SMCData.data)
float32(data:)       : Float(bitPattern: 0x0000803F)  ≈ 4.6e-41   ✗ (should be 1.0)
correct would be     : Float(bitPattern: 0x0000803F.byteSwapped = 0x3F800000) = 1.0

The same applies to ioft: ioFloat computes uint64(data) >> 16 on the reversed buffer, yielding the byte-swapped 64-bit value rather than the true one. By contrast sp78 is correct because it indexes data[1]/data[0] in a way that already accounts for the reversal (and its test feeds the post-reversal layout [0x80,0x14] → 20.5).

Impact: for every flt- and ioft-typed key — common on Apple Silicon for temperature, fan and power sensors — the decoded SMCData.value (and the smc-dump value column, via SMCDump.valueDescription) is garbage. The raw data bytes remain intact, so only the decoded value is affected. The unit tests miss this because SMCHelperTests only tests the standalone big-endian helpers, and the value() tests cover only byte-order-independent types (ui8, si8, flag, ch8*).

Recommendation

Compensate for the reversal in the flt and ioft cases exactly as the integer cases do. For example, byte-swap the assembled bit pattern before reinterpreting: 

case "flt ": return Float32( bitPattern: self.uint32( data: data ).byteSwapped )
case "ioft": return self.ioFloat( data: Data( data.reversed() ) )   // or add a byteSwapped uint64 path

Then add value(for:type:) tests that feed post-reversal (little-endian) buffers for flt, ioft, sp78, ui16/ui32/ui64 and assert the fully decoded values, so the end-to-end decode path — not just the helpers — is covered. Confirm the intended byte order against a known-good reference read on hardware.

Medium

2. The process-wide shared instance is not thread-safe

Category: stability / crash  ·  Location: SMCKit/SMC.m:218–269, 376–426

Description

SMC exposes a shared singleton intended for reuse across a process, but its mutable state is accessed without synchronization. -readAllKeys: lazily mutates self.keys (an NSMutableArray) on first use, and -readSMCKeyInfo:forKey: mutates self.keyInfoCache (an NSMutableDictionary). Both NSMutableArray and NSMutableDictionary are documented as unsafe for concurrent mutation. In addition, -callSMCFunction: opens, calls and closes the shared user-client session on the single self.connection; concurrent calls interleave open/close on the same connection.

A framework singleton is a natural thing for a client to touch from several queues (e.g. a menu-bar refresh timer plus a UI action). Two concurrent readAllKeys() / SMCDump.produce() calls can corrupt the collections or the connection state and crash.

Recommendation

Serialize access — e.g. wrap the body of readAllKeys:, readSMCKeyInfo:forKey: and the callSMCFunction: open/call/close sequence in a per-instance serial dispatch_queue_t or @synchronized(self) — and document the concurrency contract. At minimum, state in the header that SMC (including shared) is not thread-safe and must be confined to one queue.

Low

3. NSRegularExpression range uses String.count instead of the UTF-16 length

Category: correctness / performance  ·  Location: SMCKit/MachineDetails.swift:118, 124

Description

regex.matches( in: $0, range: NSMakeRange( 0, $0.count ) ) passes the Swift grapheme count as an NSRange length, but NSRegularExpression works in UTF-16 units. Because grapheme count is always ≤ UTF-16 length, this never over-runs (no crash), but for any line containing multi-unit characters (accents, emoji) the search would silently stop short and miss trailing content. It happens to be harmless today only because system_profiler SPHardwareDataType output is ASCII. Separately, the constant regex pattern is recompiled inside compactMap once per output line rather than once.

Recommendation

Use the UTF-16 length: NSRange( location: 0, length: ( $0 as NSString ).length ) (or $0.utf16.count). Hoist the NSRegularExpression out of the loop and compile it once (it is thread-safe and reusable).

Low

4. Task never removes its notification observers and may read output before it is fully delivered

Category: stability / reliability  ·  Location: SMCKit/Task.swift:98–130

Description

Task registers two NSFileHandleDataAvailable observers in its initializer but has no deinit that calls removeObserver. NotificationCenter holds observers without owning them; relying on the pipes deallocating together with the task to stop notifications is fragile. The capture design is also delivery-order sensitive: output is collected via availableData callbacks re-armed with waitForDataInBackgroundAndNotify(), while run(input:) blocks in waitUntilExit() and then reads standardOutput immediately — any final buffered chunk not yet delivered as a notification can be missed. For the small, fixed system_profiler header this works in practice, but it is brittle if reused for larger output.

Recommendation

Add a deinit that removes both observers. For robustness, prefer reading to end after exit (readDataToEndOfFile()) or draining the pipes on a dedicated queue, rather than mixing run-loop notifications with a blocking waitUntilExit() on the same thread.

Low

5. keyInfoCache leaks its heap-allocated entries

Category: memory / leak  ·  Location: SMCKit/SMC.m:416–422

Description

Each key's metadata is stored as a calloc'd SMCKeyInfoData wrapped in [NSValue valueWithPointer:]. Those raw allocations are never freed — not on cache eviction (there is none) and not in -dealloc. For the shared singleton this is a bounded one-time cost, but any client that creates and releases its own SMC instances leaks one allocation per distinct key each time.

Recommendation

Free the pointers in -dealloc (iterate keyInfoCache and free(value.pointerValue)), or avoid raw allocation entirely by boxing the struct in [NSValue valueWithBytes:objCType:@encode(SMCKeyInfoData)], which stores the value inline and requires no manual free.

Low

6. The SMC user-client session is opened and closed around every single call

Category: performance  ·  Location: SMCKit/SMC.m:349–365

Description

-callSMCFunction:input:output: issues kSMCUserClientOpen before, and kSMCUserClientClose after, each structured call. A full readAllKeys() performs at least two such calls per key (key-info + read) across hundreds of keys, so the enumeration does many hundreds of redundant open/close round-trips into the kernel. Functionally correct, but avoidable overhead on a hot path.

Recommendation

Open the user-client session once around a batch (e.g. for the duration of readAllKeys:) rather than per call, or verify whether the open/close is required at all for kSMCHandleYPCEvent. Keep whatever pairing is correct, but hoist it out of the innermost call.

7. calloc arguments are in the wrong order

Category: style / correctness  ·  Location: SMCKit/SMC.m:416

Description

calloc( sizeof( SMCKeyInfoData ), 1 ) passes the element size as the count and 1 as the size. The total byte count is identical so it is harmless, but the intent is reversed and reads incorrectly.

Recommendation

Write calloc( 1, sizeof( SMCKeyInfoData ) ) to match the calloc(count, size) signature.

8. -close returns lowercase false amid YES/NO

Category: style  ·  Location: SMCKit/SMC.m:331

Description

The method returns false on the failure path while every other return in the file uses the Objective-C YES/NO spelling. Same value, inconsistent convention.

Recommendation

Use NO for consistency.

9. flag decoding treats only exactly 1 as true

Category: correctness (minor)  ·  Location: SMCKit/SMCHelper.swift:82

Description

case "flag": return self.uint8( data: data ) == 1 ? true : false maps any non-1 byte (including 2, 0xFF) to false. SMC flags are conventionally 0/1 so this is fine in practice, but a defensive != 0 better matches boolean intent.

Recommendation

Consider self.uint8( data: data ) != 0.

Info

10. fourCC can embed a NUL for zero bytes

Category: robustness (minor)  ·  Location: SMCKit/SMCHelper.swift:44–53

Description

String( format: "%c%c%c%c", … ) renders each byte as a character; a zero byte produces an embedded NUL in the resulting String. Key and type codes are normally printable ASCII, so this rarely bites, but a key/type containing a 0x00 byte yields a string with an embedded null that can confuse downstream display, sorting, or comparison.

Recommendation

If robustness for arbitrary codes matters, map non-printable bytes to a placeholder (as printableString already does) rather than emitting raw control characters.

Strengths

  • Clear, consistent structure: one type per file, sensible Objective-C/Swift split, and a dedicated SMCException bridge so Swift can recover from Objective-C exceptions thrown by Process.
  • Thorough, accurate documentation throughout (HeaderDoc for the Objective-C surface, SwiftDoc for the Swift surface), including private members.
  • Defensive size checks in every integer/float decoder (guard data.count == N else { return 0 }) prevent out-of-bounds reads on malformed buffers, and readSMCKey:buffer:maxSize: guards the destination buffer against oversized values.
  • Good use of value semantics and immutability (let, private(set)), no force-unwraps, and functional iteration (map/reduce/forEach) consistent with the project's conventions.
  • Newly added Swift Testing coverage for the decoding helpers and SMCData — a solid base to extend to the end-to-end value() path (see finding 1).
