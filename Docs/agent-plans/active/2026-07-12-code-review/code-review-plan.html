Overview

This plan turns the ten findings from the SMCKit code review into small, individually reviewable and committable milestones. The review rated the codebase's overall quality as good; the work here is targeted remediation rather than a rewrite. The findings span one high-severity value-decoding correctness bug, one medium-severity thread-safety gap, and a set of low/info robustness, leak, performance, and style items.

The milestones are grouped into four phases: Correctness (the flt/ioft byte-order bug), Stability & memory (thread-safety, the key-info cache leak, and the per-call open/close), Robustness (the system_profiler regex and the Task output capture), and Minor correctness & style. The headline byte-order bug is deliberately split into a hardware-verification milestone that gates the actual code change, because the correct byte order must be confirmed against a known-good read on real hardware before any decode logic is altered. Per the project conventions, the Xcode project and the Swift package are both kept in sync, and tests are run through xcodebuild against the SMCKit scheme.

Questions & answers

Q. Finding 1 (flt/ioft byte order) is the headline correctness bug and the review recommends confirming the intended byte order against a known-good hardware read, which cannot be done in this environment. How should the plan treat it?

A. Hardware-verify first: add a milestone to confirm the byte order on real hardware before changing any decode logic, and treat the fix as blocked until that verification is done.

Q. Finding 2 is the process-wide shared instance not being thread-safe. Which approach should the plan take?

A. Serialize access — wrap readAllKeys:, readSMCKeyInfo:forKey: and the callSMCFunction: open/call/close sequence in a per-instance serial queue (or @synchronized), and document the concurrency contract.

Q. Findings 4 and 6 are performance/robustness items the review itself flags as uncertain or hardware-dependent. Include them in the plan?

A. Include all — add milestones for findings 4 and 6 alongside the rest, treating finding 6 carefully given its hardware-dependent behavior.

Q. Overall, which findings should the plan cover?

A. All findings (1–10), grouped into reviewable milestones, subject to the per-finding decisions above.

Notes

  • Byte-order model. SMC.m -readSMCKey:buffer:maxSize:keyInfo: (lines 528–538) reverses every value's bytes from the SMC's big-endian ordering into little-endian before handing them to Swift, with kSMCKeyACID as the sole exception. The integer cases in SMCHelper.value(for:type:) compensate with .byteSwapped; sp78 compensates via its indexing; flt/ioft currently do not — that is finding 1.
  • Helpers vs. the value() path. The standalone numeric helpers (uint16, float32, ioFloat, …) are documented and tested as big-endian decoders and are internally consistent. Existing tests in SMCHelperTests cover the helpers and the byte-order-independent value() cases only; no test exercises value(for:type:) for the reversed multi-byte types.
  • Key files. SMCKit/SMCHelper.swift (decoders), SMCKit/SMC.m (IOKit access, cache, byte reversal), SMCKit/MachineDetails.swift (regex), SMCKit/Task.swift (process output), SMCKit/SMCData.swift and SMCKit/SMCDump.swift (value presentation), tests in SMCKitTests/.
  • Conventions. One type per file, no force-unwraps, prefer let, prefer map/filter/reduce over raw loops, explicit public in the framework target, and doc comments on private members too. Keep the .xcodeproj and the SwiftPM manifest in sync; build and test via xcodebuild -project SMCKit.xcodeproj -scheme SMCKit.
  • Hardware dependency. Milestone 1 and the on-hardware validation of Milestone 5 must be run by the maintainer on a real Mac; they cannot be verified in a sandbox. Each is called out explicitly.

1. Verify the flt/ioft byte order on real hardware

Goal: Establish, against a known-good reference, whether flt and ioft values need byte-swap compensation before the decode logic is touched. This milestone gates Milestone 2 and must be run on a real Mac — it produces evidence, not code.

Tasks
  • Identify one or more live keys of type flt and ioft on the target hardware (e.g. an Apple Silicon temperature or power sensor).
  • Read each key via SMC.shared.readAllKeys() and capture, per key: the raw post-reversal SMCData.data bytes, the current decoded SMCData.value, and the value smc-dump prints.
  • Obtain an independent reference value for the same sensors (e.g. powermetrics, ioreg, or a trusted third-party monitor) to compare against.
  • For at least one sample, compute both the plain and the byteSwapped interpretations of the raw bytes and record which one matches the reference.
  • Record the conclusion in this milestone's Implementation notes: the exact fix required for flt and for ioft (byte-swap or not), plus the concrete byte/value samples used to derive expected values for the Milestone 2 tests.

Status: Completed   ·   Completed: 2026-07-12

Implementation notes

Verification environment. MacBook Air, Mac16,12, Apple M4 (arm64), macOS 26.5.1 (25F80). Evidence gathered read-only by building the smc-dump scheme (xcodebuild -project SMCKit.xcodeproj -scheme smc-dump) and dumping all live keys. The dump exposes, per key, the raw post-reversal SMCData.data bytes (hex) and the value produced by the current (unmodified) decode path. This machine reports 993 flt keys and 12 ioft keys.

Headline result — Finding 1 is a false positive on this hardware. The current flt/ioft decoding already yields physically-correct values, and the byte-swap compensation the review recommends (Milestone 2) would corrupt them. No code change to the decode logic is warranted for flt or ioft.

Raw byte samples — current decode vs. the proposed .byteSwapped fix:

key   type  raw (post-reversal)  current decode      proposed .byteSwapped
----  ----  -------------------  ------------------  ---------------------

TCHP  flt   423BD038             46.95  (temp °C)    9.9293e-05   garbage
TDBP  flt   42100000             36.00  (temp °C)    5.8322e-42   garbage
Bvt7  flt   408B31A8             4.35   (voltage)   -9.8557e-15   garbage
ceU0  flt   42C80000             100.00 (percent)    7.1839e-41   garbage

TG0H  ioft  0000000000220000     34.00  (temp °C)    5.7043e+08   garbage
TG0V  ioft  0000000000229999     34.60  (temp °C)    1.6888e+14   garbage
TR0Z  ioft  000000000033D1EC     51.82  (temp °C)    2.6038e+14   garbage

Independent cross-checks (the "known-good reference"):

  • Physical plausibility. Across ~1000 flt keys the current values sit in sane ranges: chip-temperature sensors (TB0T, TCHP, TCMz, TDBP, the Te0x/Tg0x families) read 33–79 °C; battery/rail voltages ~4.35 V; percentages 0/100; and disabled sensors report the conventional -50000 sentinel. The .byteSwapped interpretation instead produces IEEE denormals (~1e-41) or absurd magnitudes (1e+22), i.e. it is self-evidently wrong.
  • Two independent decode paths agree. TB0T (flt) and TG0V (ioft) — different code paths, different buffer widths — both independently decode to 34.60 °C with the current logic. Under the proposed fix they diverge to 1.7e+22 and 1.7e+14 respectively.

Why the reviewer flagged it, and why the asymmetry is correct. The finding rests on a code-symmetry argument: the integer cases apply .byteSwapped after SMC.m's per-value byte reversal, so flt/ioft "should" too. Empirically they must not — on this SMC the integer and float families require opposite treatment after the reversal, and the library historically shipped the flt/ioft cases without a swap. The live data confirms the original author's behavior is the correct one.

Confirmed fix required. flt: none. ioft: none. The decode logic in SMCHelper.value(for:type:) is correct as-is for both types on Apple Silicon.

Scope / caveat (Intel). Verified only on Apple Silicon (M4). The Swift decoders build integers explicitly from individual bytes and .byteSwapped operates on the value, so all of this is host-endianness-independent (holds identically whether the host is arm64 or x86_64); the byte layout being decoded is a property of the SMC wire format, not the CPU. Intel Macs use a physically different SMC, which this environment cannot exercise, but the current code has shipped against Intel hardware historically. Recommend a maintainer confirm one flt and one ioft key on an Intel Mac if one is available before closing the finding across the board.

Consequence for Milestone 2. Milestone 2 as written ("apply the byte-order compensation … Float32(bitPattern: uint32().byteSwapped) for flt, byte-swapped path for ioft") should not be implemented — it would break correct decoding. It should be re-scoped to lock in the verified-correct behavior with regression tests (feeding these real post-reversal buffers through value(for:type:) and asserting the current values), plus a doc note explaining the intentional int-vs-float asymmetry so this is not "fixed" again. Awaiting maintainer decision.

Completed Finding 2 · Medium

3. Serialize the process-wide shared SMC instance

Goal: Make concurrent use of an SMC instance (especially shared) safe by serializing all access to its mutable state and its user-client session, and document the concurrency contract.

Tasks
  • Add a per-instance serial dispatch_queue_t (or use @synchronized(self)) in SMCKit/SMC.m, created in -init.
  • Serialize the bodies that touch shared mutable state: -readAllKeys: (lazy mutation of self.keys), -readSMCKeyInfo:forKey: (mutation of self.keyInfoCache), and the open/call/close sequence in -callSMCFunction:input:output: so concurrent calls cannot interleave on the single self.connection.
  • Avoid re-entrant deadlock: ensure a serialized public method does not synchronously call another method that acquires the same lock/queue (e.g. keep the synchronization at a single, outer boundary, or use a recursive strategy).
  • Document the concurrency contract in the header (SMC.h): state that access is serialized internally, or — at minimum — that SMC including shared must be confined appropriately.
  • Build and run the existing suite; confirm no regression. (Add a lightweight concurrent smoke test only if it can run without real SMC hardware; otherwise note manual verification.)

Status: Completed   ·   Completed: 2026-07-12

Implementation notes

Approach — serial queue at a single outer boundary. Used a per-instance serial dispatch_queue_t queue created in -init rather than @synchronized. Rationale: readAllKeys: is the only public multi-step entry point (SMC.h exposes just +shared, -init, -readAllKeys:), and every SMC access funnels through its deep call chain (readAllKeys: → readSMCKeyCount / readSMCKey:atIndex: / readSMCKey:buffer:maxSize:keyInfo: → readSMCKeyInfo:forKey: / callSMCFunction:input:output:). Serializing at that single boundary transitively serializes all access to keys, keyInfoCache and connection — the "single, outer boundary" the plan explicitly permits.

  • SMCKit/SMC.m — added the serial queue property (created in -init). Kept the original readAllKeys: body verbatim as a new private readAllKeysUnsynchronized: worker; readAllKeys: is now a thin wrapper that dispatch_syncs onto queue and delegates. No nested dispatch_sync exists anywhere in the chain, so re-entrant deadlock is impossible.
  • open:/close left unsynchronized by design: both are private and mutate connection only from -init/-dealloc (lifecycle-bound — an instance cannot be deallocated while another thread is executing one of its methods). See the deferred Info note below.
  • SMCKit/SMC.h — documented the concurrency contract on the class discussion and on readAllKeys:; the internal queue and worker doc comments state the serialization/queue-affinity rules.
  • SMCKitTests/SMCTests.swift — new SMC suite with a concurrency regression test hammering SMC.shared.readAllKeys() from 16 concurrent tasks and asserting all complete. Hardware-independent (returns empty arrays where no SMC connection exists, so it is CI-safe) yet exercises the serialized paths. Registered in SMCKit.xcodeproj (PBXBuildFile / PBXFileReference / group / Sources phase, fresh non-colliding IDs). No SwiftPM manifest exists, so only the Xcode project needed updating.

TDD via Thread Sanitizer (real red→green). A plain test cannot deterministically fail on a data race, so the concurrency test was run under TSan against the current code: TSan reported a genuine data race on NSMutableDictionary (-[__NSDictionaryM setObject:forKeyedSubscript:], i.e. self.keyInfoCache) — red. After adding the serialization the same test under TSan reported 0 data races — green.

Verification. Full suite xcodebuild -project SMCKit.xcodeproj -scheme SMCKit test — 24/24 passed (23 existing + the new concurrency test), no regressions. Clean build with the project's -Wdocumentation -Werror after the doc edits. Temporary build artifacts removed. Left the lowercase return false; in -close untouched — it belongs to Milestone 9.

No Critical/High/Medium findings.

5. Hoist the user-client open/close out of the per-call hot path

Goal: Avoid the hundreds of redundant kSMCUserClientOpen/kSMCUserClientClose round-trips performed during a full readAllKeys(), without changing observable behavior. This is hardware-dependent and must be validated on a real Mac.

Tasks
  • Investigate whether the open/close pairing around each IOConnectCallStructMethod is actually required for kSMCHandleYPCEvent, or whether the session can be opened once around a batch. Record the finding.
  • If safe, open the user-client session once for the duration of a batch (e.g. around -readAllKeys:) rather than per call, keeping whatever pairing is correct but out of the innermost call. Coordinate with the Milestone 3 serialization so the session lifetime stays consistent.
  • If open/close per call turns out to be required, keep it and instead document why, closing out the finding without a risky change.
  • Validate on real hardware that readAllKeys() still returns correct values and that smc-dump output is unchanged.
  • Build and run the suite.

Status: Completed   ·   Completed: 2026-07-12

Implementation notes

Hardware investigation (Apple Silicon M4, macOS 26.5.1). Captured a baseline smc-dump (1979 keys read). Then, as an experiment, removed the per-call kSMCUserClientOpen/kSMCUserClientClose entirely from callSMCFunction: (relying only on the IOServiceOpen connection) and re-dumped: identical — same 1979 keys and a byte-for-byte identical key/type/size set (empty diff on the stable columns), with correct values. So on this hardware the per-call session open/close is not functionally required; the IOServiceOpen connection alone suffices.

Design chosen — hoist, don't remove (Intel-safe). Because Intel Macs use a physically different SMC that cannot be exercised here and may genuinely need the session, the pairing was kept but moved out of the per-call hot path to once per readAllKeys batch — the finding's literal recommendation. Removing it entirely was rejected as too risky for hardware that cannot be tested here.

  • callSMCFunction: no longer opens/closes; it now just performs the IOConnectCallStructMethod. Doc updated.
  • Added two private helpers: openUserClientSession (sends kSMCUserClientOpen, returns success) and closeUserClientSession (sends kSMCUserClientClose), with HeaderDoc and private-category declarations.
  • readAllKeysUnsynchronized: now opens the session once (after the null-connection guard; returns @[] if the open fails), runs the full enumeration + read loop, then closes it before returning. This runs entirely on the Milestone 3 serial queue, so the session calls stay serialized with all other connection access — no new race, no change to the concurrency contract. Per-batch open/close drops from ~thousands to exactly one pair.

Validation. Rebuilt smc-dump with the shipping design and re-dumped: identical to baseline (1979 keys, empty triple diff, correct values). Full suite xcodebuild -scheme SMCKit test — 24/24 passed. Clean build under -Wdocumentation -Werror. All experiment builds cleaned up; the working tree holds only SMCKit/SMC.m. Left the lowercase return false; in -close untouched (Milestone 9).
